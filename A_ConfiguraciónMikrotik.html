    Mikrotik Hardening.¶

    Vamos a ver las políticas de seguridad más habituales que podemos configurar en los equipos de red (hardening). Practicaremos con el mikrotik y posteriormente buscaremos opciones para otros fabricantes.

    Mantener el Mikrotik actualizado.¶

    Crear un nuevo usuario administrador:¶

    1. Crear un nuevo usuario/contraseña administrador (full).

    2. Si es posible delimitar una red de mantenimiento. La gestión del equipo solo se podrárealizar desde un equipo con IP de dicha red.

    3. Borrar o desactivar el anterior administrador (por defecto).

    Deshabilitar servicios que no usaremos.¶

    1. Si solo vamos a hacer uso de una gestión a través de “ssh” y “winbox” no tiene sentido tener habilitados otros servicios. Por supuesto servicios como “telnet”, “ftp” o “www” deberían ser directamente evitados.

    2. Cambiar los puertos habituales (ssh:22 y winbox: 8921).

    3. Filtrar por red de “mantenimiento”.

    Acordaros que si cambiamos el puerto, desde el cliente tenemos que indicárselo

    4. En el caso de Mikrotik (no es habitual en otros fabricantes), se puede acceder a través de la MAC. Para ello tiene unos servicios (MAC Server), que por norma general se deben desactivar por seguridad:

    Deshabilitar Protocolos.¶

    Existen algunos protocolos, habitualmente propietarios, que permiten descubrir o hacerte una idea de los equipos existentes en la red. Esto también puede dar información a los atacantes, además de que muchas veces estos protocolos no tienen ningún uso por parte de los administradores. En el caso de Mikrotik es “Neighbor Discovery”, en el caso de Cisco existe CDP (Cisco Discovery Protocol).

    Desactivar las interfaces que no vayan a usarse.¶

    Reglas de Firewall para el plano de control (chain: input).¶

    Permitir pasar las conexiones ya establecidas/relacionada. Es decir, cada vez que una conexión con el router se establece sí se compara con las reglas de tu firewall, pero una vez la conexión ya esté establecida (TCP) ya no contrastamos con ninguna regla. Esta decisión alivia la carga del firewall. Es importante que la regla esté situada arriba del todo para que no mire el resto de normas.

    Una conexión relacionada es un tráfico relacionado a una configuración ya establecida, por ejemplo el tráfico FTP relacionado con la sesión FTP establecida.

    add action=accept chain=input connection-state=established,related

    Permitir que solo tráfico originado desde la red de mantenimiento vaya a ser atendida por el router. Para ello es recomendable crear un Address-list con los rangos de mantenimiento.

    Warning

    Es importante al final bloquear el resto de tráfico. En ciertos frabricantes esta regla viene implícita, pero no es una mala práctica.

    add action=drop chain=input

    Reglas de Firewall para el plano de datos (chain: forward).¶

    Al igual que hacíamos, la primera regla con el fin de aliviar la carga del firewall es permitir pasar las conexiones ya establecidas. La acción “fasttrack-connection” es una opción especial de Mikrotik para detectar de forma rápida que tráfico procede de una conexión ya establecida correctamente.

    add action=fasttrack-connection chain=forward connection-state=established,related
add action=accept chain=forward connection-state=established,related

    Por otro lado bloqueamos las conexiones TCP inválidas (ataque TCP), estas conexiones son aquellas donde se intenta enviar tráfico sin haber establecido previamente una conexión. Como una conexión inválida es posible que se deba a un ataque, le habilitamos el log=yes para que lo deje registrado, además lo marcamos con la etiqueta “invalid”. 

    add action=drop chain=forward connection-state=invalid log=yes log-prefix=invalid

    Bloquear tráfico que salga de nuestra LAN hacia la WAN cuya IP de destino sea una IP no válida en Internet (por ejemplo una IP privada).

    add action=drop chain=forward dst-address-list=not_in_internet in-interface=bridge1 outinterface=ether1 log=yes log-prefix=!public_from_LAN

    Tip

    Para el ejemplo anterior se hace uso de un address-list llamada "not_in_internet" que acoge todas las redes privadas y reservadas.

    Si nuestra LAN se encuentra tras un NAT, bloquearemos cualquier tráfico que tenga como destino las IPs sin NATEAR desde fuera de nuestra red (WAN).

    add action=drop chain=forward connection-nat-state=!dstnat connection-state=new ininterface=ether1 log=yes log-prefix=!NAT

    Bloquear todo el tráfico que venga desde nuestra WAN con IP origen una IP no válida en Internet.

    add action=drop chain=forward in-interface=ether1 src-address-list=not_in_internet log=yes log-prefix=!public

    Para limitar el efecto de IP spoofing, podemos crear un filtro donde solo permitamos tráfico desde nuestra LAN con IP origen de las redes que tenemos en ella asignadas.

    add action=drop chain=forward in-interface=bridge1 src-address-list=LAN log=yes logprefix=LAN_!LAN

    Si dentro de nuestra red no existe, o no debería existir ningún servicio DNS o web, podría ser una buena política cortar el tráfico entrante a nuestra red para los puertos 53 y 80. De esta forma evitamos que si por alguna casualidad alguien levanta un servicio DNS en nuestra red (sin nuestro conocimiento) éste pueda ser atacado.

    Hemos aplicado estas reglas en el apartado de RAW por motivos de eficiencia, ya que en este apartado podemos crear filtros prerouting, es decir se ejecutan antes de comparar la tabla de rutas y saber porque interfaz deben ir dirigidos.
