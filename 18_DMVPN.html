    DMVPN (Dynamic Multipoint VPN).¶

    Es un protocolo creado por Cisco, no obstante se basa en 3 protocolos no propietarios, por lo que es factible la implementación de un protocolo similar por otros fabricantes.

    La idea detrás de DMVPN es generar conexiones seguras VPN de forma dinámica, habitualmente entre un concentrador (HUB) y varios nodos (SPOKE).

    Protocolos.¶

    NHRP¶

    • Permite a los SPOKE descubrir el HUB.

    • Permite transportar tráfico multicast/broadcast a través del túnel como si se encontraran en la misma red, lo que nos permite que protocolos de enrutamiento dinámico funcionen entre los SPOKE y el HUB.

    Protocolo de enrutamiento dinámico (EIGRP/OSPF…).¶

    • Transfieren sus tabla de rutas para alcanzar las redes conocidas.

    • Permite configurar caminos alternativos ante caídas de servicios/conexiones.

    • Permite a los SPOKE comunicarse a través de túneles independientes de forma directa.

    MultiPoint GRE (mGRE).¶

    • Protocolo que establece los túneles entre los SPOKE y el HUB.

    • Dentro de este túnel realiza sus tareas el NHRP para pasarse la información de configuración de IPSec, de los protocolos de enrutamiento dinámico, etc.

    Proceso.¶

    1. Crear Túnel Gre

    Cuando levanta un SPOKE, tiene configurada la IP destino del HUB (IP fija) dentro de la interfaz túnel, mediante el protocolo NHRP (para informar al HUB de quien está intentando levantar un túnel) es capaz de levantar un túnel GRE con el HUB.

    2. Establecer IPSec

    Con el túnel GRE ya levantado el HUB es capaz de enviar a través de NHRP los datos para la configuración del protocolo IPSec.

    3. Establecer Protocolo de Enrutamiento Dinámico

    Una vez establecido el túnel GRE junto con IPSec se manda, a través de NHRP, el tráfico multicast referente a los protocolos de enrutamiento dinámico. El tráfico multicast debe enviarse por NHRP puesto que IPSec no soporta el cifrar este tipo de tráfico.

    A través de los protocolos de enrutamiento dinámico cada SPOKE sabrá las redes LAN de cada una de ellas y se permitirá comunicación directa SPOKE-to-SPOKE, en caso de ser necesario.

    Ventajas DMVPN.¶

    Configuración más sencilla y muchísimo más escalable.

    💥 En caso de NO tener DMVPN, tendríamos que crear un túnel de forma estática en la configuración para cada sede con la que queramos establecer una VPN, configurando dentro de este túnel todo lo relativo a IPSec para que sea seguro.

    💥 Aproximadamente este tipo de configuración en el HUB lleva al menos unas 13 líneas por SPOKE, por lo que si tuviéramos 300 SPOKES serían 3900 líneas de código de configuración… ➡️ Si aplicamos DMVPN la configuración total del HUB con 300 SPOKES sería de aproximadamente unas 20 líneas.

    Permite que los SPOKE tengan configurada una IP dinámica por su ISP.

    💥 Dependiendo de la configuración se puede permitir que los SPOKE tengan asignada una IP dinámica.

    💥 Para establecer el túnel el SPOKE buscaría la IP del HUB (ésta sí debe ser fija) y mediante NHRP le informaría de su nueva IP para establecer el túnel.

    Permite la comunicación directa entre SPOKES.

    💥 En caso de no usar DMVPN, tendríamos que establecer los túneles entre el HUB y el SPOKE a mano.

    💥 En caso de que un SPOKE quisiera hablar con otro SPOKE tendría que enviar el tráfico al HUB, que éste descifrara el tráfico, lo procesara y lo volviera a cifrar para enviárselo al SPOKE destino. Este proceso sobrecarga la CPU del HUB, que en caso de tener muchos túneles puede ser significativo, además de que obviamente se añadiría un poco de latencia.

    Ejemplo de Configuración.¶

    Configuración del IPSec.¶

    • Authentication pre-share ➡️ Se utilizará clave compartida (PSK) en vez de por ejemplo certificados digitales (RSA).

    • Crypto isakmp key XXXXX address 0.0.0.0 0.0.0.0 ➡️ Como los SPOKES tendrán IPs dinámicas, se permite establecer IPSec con cualquier IP (0.0.0.0).

    • Crypto ipsec transform-set trans2 esp-des esp-md5-hmac mode transport ➡️ Se crea un perfil “trans2” que usará “ESP” + “DES” + “md5” + modo transporte. En el caso de querer especificar un grupo de Diffie-Hellman tendríamos que especificarlo con el comando “group X”.

    SEGURIDAD

    Esta configuración de ejemplo NO ES SEGURA, ya que se debe evitar el uso de DES y md5 en especial.

    Configuración del Túnel GRE.¶

    Redes dentro del túnel

    En los túneles GRE se deben de especificar 2 redes:

    💥 La red que realmente crea el túnel y que debe estar configurada en alguna interfaz y ser alcanzable por el otro extremo (Cabecera IP lila en la imagen).

    💥 La red que compone el túnel en si (Cabecera IP verde en la imagen).

    • tunnel mode gre multipoint ➡️ Generaremos el túnel GRE, aunque al no ser un túnel, si no varios (del HUB a muchos SPOKES), se usa el protocolo mGRE (multipoint GRE).

    • tunnel source Ethernet0 ➡️ Es la IP de la red física a través de la cual se establece el túnel. También se podría especificar una IP en vez de la interfaz.

    • tunnel destination 172.17.0.1 ➡️ Es la IP física del destino contra el cual se establece el túnel. Sólo se configurará en los SPOKE.

    • ip address IP MÁSCARA ➡️ Son las IPs lógicas que conformarán el túnel.

    Enrutamiento

    Si queremos que el tráfico vaya por el túnel lo enrutaremos por las IPs lógicas que conforman el túnel.

    Entendemos por interfaces lógicas aquellas que son virtuales (no tienen una interfaz física en el equipo) y una IP lógica serán aquellas redes asignadas a dichas interfaces.

    • ip hrp authentication XXXX ➡️ En caso de querer ponerle una contraseña al protocolo NHRP. Es independiente del IPSec y debe coincidir en ambos extremos.

    • ip nhrp multicast dynamic ➡️ Permite transportar tráfico multicast, esto nos sirve para establecer protocolos de enrutamiento dinámico, los cuales usan direcciones multicast para comunicarse el enrutamiento.

    • ip nhrp network-id XXXXX ➡️ Habilita NHRP en el túnel, le asigna un identificador.

    • Tunnel key XXXX. ➡️ En el caso de DMVPN es necesario darle un identificador al túnel.

    • Tunnel protection ipsec profile vpnprof ➡️ Asigna el perfil IPSec al túnel.

    Troubleshooting.¶

    A continuación algunos comandos para depurar si tenemos todo correcto.

    • Comprobar si el túnel y las interfaces físicas están levantadas.
    • Mirar si el IPSec está activo.
    • Ver si el protocolo de enrutamiento dinámico está levantado.
    • Averiguar qué rutas conozco por el protocolo de enrutamiento, en el caso del ejemplo EIGRP.
