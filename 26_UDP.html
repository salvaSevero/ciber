    DDoS with UDP flooding.¶

    UDP

    Uno de los ataques más básicos es aprovecharse del protocolo UDP, el cual NO está orientado a conexión como TCP, y por tanto no hay que esperar ningún acuse de recibo ACK para seguir mandando paquetes, no hace falta establecer una conexión previa, etc. Es decir, podemos enviar todo el tráfico que deseemos de golpe.

    El ataque consistuiría en una red de bots que lanzaran a la vez un ataque UDP flooding al objetivo.

    Características del ataque.¶

    • Cada bot/PC infectado solo podrá enviar tanto tráfico como su línea de Internet/interfaz de red le permita.

    • Podemos hacer IP spoofing para intentar guardar la identidad de nuestros PCs infectados, no obstante, si hacemos esto hay muchas posibilidades de que estas IP spoofeadas sean cortadas por algún cortafuegos.

    • El ataque con UDP se manda a un puerto aleatorio, además de consumir el ancho de banda de la víctima, esta al ver que en ese puerto no hay ninguna aplicación escuchando devolverá un ICMP Destination Unreachable. Por lo que también saturaremos/deterioraremos su CPU.

    Mitigación del ataque:¶

    • Configura el servidor para que no conteste con el protocolo ICMP (uso de discard).

    • Utilizar un cortafuegos que cuando detecte un patrón de tráfico UDP de este estilo con gran cantidad de tráfico lo corte automáticamente.

    Escenario

    Atacante (Windows) ➡️ WAN MiKrotik LAN ➡️ Víctima (Windows/Linux).

    PASO 1. Comprobar Conectividad

    Comprobar que hay conectividad (ping) entre el atacante y la víctima. El ping no es necesario, simplemente lo hacemos para cerciorarnos que el laboratorio está bien conectado y no dar palos de ciego si luego falla algo. Es muy probable que tengamos que deshabilitar algún firewall de Windows/Linux para comprobar la conexión. Una vez comprobado se pueden volver a habilitar.

    Mitigación.¶

    En caso de que no dispongamos de un equipo firewall o antiDDoS configurado para cortar este tipo de ataques de forma automática podríamos hacer lo siguiente:

    • Disponer de un sistema de monitorización de alertas: Cacti, Nagios, Zabbix, Icinga, etc… que nos avise de que hay un tráfico anómalo.

    • Usar algún tipo de sniffer dentro de los routers, por ejemplo la herramienta Torch dentro de Mikrotik.

    Herramienta Torch.¶

    Con la herramienta torch podemos seleccionar la interfaz en la que queremos escuchar, podemos filtrar por IPs (origen/destino), VLANs, protocolos etc.

    Anexo.¶

    Ruta estática en Windows.¶

    Si necesitáis crear una ruta estática en vuestra máquina anfitriona para llegar a la LAN virtual:

    • 192.168.88.0/24 es la red LAN a la que quiero llegar.

    • 192.168.1.140 es la IP de la WAN que le han asignado al Mikrotik.

    route add 192.168.88.0 MASK 255.255.255.0 192.168.1.140

    Limitación Mikrotik Virtual.¶

    El mikrotik virtual tiene una licencia de prueba que limita a 1Mbps el tráfico que puede sacar. Por eso no vemos que por la interfaz LAN pueda sacar más de 1Mbps pese a que el ataque sea mayor.

    Destination Unreachable.¶

    Se puede filtrar por icmp para ver únicamente los Destination recibidos.

    También se puede revisar la carga de la propia CPU, ésta será muy diferente si el ataque va a un cliente (plano de datos) o contra el propio router (plano de control).
