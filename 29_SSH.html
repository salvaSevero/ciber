    SSH (Secure Shell Protocol).¶

    • Protocolo para el acceso remoto y posterior gestión de equipos.
    • Puerto usado por defecto 22 TCP.
    • Versión actual 2.0.

    Protocolos.¶

    Se basa en 3 tipos de protocolos.

    Cifrado Asimétrico.¶

    • Se usa un par de claves (pública/privada).
    • Es computacionalmente muy costoso para cifrar/descifrar todos los mensajes de las comunicaciones. Para aliviar este problema usamos el cifrado simétrico.

    Cifrado Simétrico.¶

    • Se usa una clave compartida.
    • No están seguro como cifrado asimétrico.
    • Es más rápido computacionalmente que el cifrado asimétrico a la hora de cifrar/descifrar cada mensaje.
    • Su punto débil es a la hora de intercambiar la clave para cifrar/descifrar, para evitar esta debilidad usamos el cifrado asimétrico y un algoritmo de intercambio de claves.

    Hashing.¶

    • Permite a partir de una entrada (mensaje, fichero, …) generar una cadena de caracteres.
    • Es casi imposible volver atrás, es decir, de la cadena de caracteres volver a generar el mensaje/fichero, por eso se conocen como funciones One-Way.
    • Si se usa la misma clave hash sobre el mismo fichero/mensaje se obtiene siempre la misma cadena de caracteres (cadena/código hash). Este tipo de estrategias, podemos verlas, con sus diferentes variantes al establecer sistemas de seguridad como VPNs o SSL/TLS.

    Fingerprint (Huella Digital).¶

    • Cuando nos conectamos a un servidor por SSH lo primero que sucede es que nos transfiere su clave pública.
    • El fingerprint se obtiene de la clave pública y obviamente es único para cada clave.
    • Es lo primero que nos aparece cuando nos conectamos por primera vez a un servidor SSH.
    ¿Cuál es su función?
    • Nos informa de que nos estamos conectando a un servidor cuyo identificador viene dado por la clave hash indicada más arriba “Xpoy95…”.

    • Si la próxima vez que nos conectemos al mismo servidor nos vuelve a salir el aviso con un nuevo identificador quiere decir que algo ha cambiado:

      💥 Alguien se está haciendo pasar por nuestro servidor.

      💥 Ha habido algún problema en el servidor y por algún motivo se han borrado todas las claves generadas.

    ¿Qué hacen los clientes con esta información?

    Los clientes almacenan la clave pública para compararla en próximas conexiones, es decir, si en las siguientes conexiones al servidor 10.0.0.1 del ejemplo, el fingerprint es el mismo no nos saldrá ningún mensaje de aviso. Recordemos que el fingerprint se puede obtener de la clave pública.

    Algoritmo Diffie-Hellman.¶

    Diffie-Hellman sólo se usa para el intercambio de claves, NO se usa para autentificar

    ¿Por qué no usamos directamente RSA para toda la comunicación?
    • Es muchísimo más ineficiente y costos computacionalmente.
    • RSA no permite PFS (Perfect Forward Secrecy). Básicamente va modificando la clave simétrica cada cierto tiempo, de forma que si se descubre dicha clave no pueden descifrar todos nuestros mensajes pasados. En caso de tener capturado nuestro tráfico con RSA y en algún momento perder o romper la clave tendrían acceso a cualquier comunicación en el tiempo que hubiese sido capturada.
    • Necesita de padding (añadir texto aleatorio antes de encriptar para aumentar la aleatoriedad) para hacerlo seguro.

    Definitiva, al ser mucho más costoso computacionalmente se utiliza la base del cifrado asimétrico para crear una clave simétrica compartida, con la cual vamos a cifrar los mensajes. Esta clave servirá para cifrar y descifrar (cifrado simétrico).

    El algoritmo que se encarga de crear una clave compartida (simétrica) sin que viaje por Internet se conoce como algoritmo de Diffie Hellman.

    Funcionamiento¶

    1. Ambos (emisor y receptor) acuerdan en un “valor semilla” que no es más que un número primo muy largo.¶

    #Ejemplo
A y B acuerdan un número (semilla) --> 17

    2. Ambos aplican mecanismos de encriptación sobre dicha semilla para generar un nuevo número.¶

    ¿Qué mecanismos de encriptación?

    Unos acordados al principio de la conexión.

    #Ejemplo
F(17) --> 4

    3. Creación clave privada.¶

    De forma independiente, cada uno genera otro número primo aleatorio desconocido para el otro extremo (clave privada).

    #Cada extremos genera un número primo aleatorio (clave privada).
A --> 3
B --> 5
    ¿Los extremos comparten dicha clave privada?

    No, nunca se debe compartir la clave privada.

    4. Creación de la clave pública.¶

    Cada host realiza una operación matemática combinando:

    • La semilla (número primo).

      💥 Ejemplo: A y B (17).

    • Nuevo número generado en el apartado b.

      💥 Ejemplo: A y B (4)

    • La clave privada generada (número primo aleatorio).

      💥 Ejemplo: A (3) y B (5).

    #Ejemplo
A --> 4^3 mod 17 = 13
B --> 4^5 mod 17 = 4

    El resultado es otro número que será la clave pública de dicha comunicación y que se intercambiaran.

    Creación clave simétrica.¶

    • Cada extremo recibe la clave pública del punto anterior (A:13 y B:4).

    • Aplicando una función matemática usando como operandos:

      💥 Las claves privadas ➡️ A (3) y B (5).

      💥 La clave pública ➡️ A (4) y B(13)

      💥 La semilla (número primo) ➡️ A y B (17)

    #Ejemplo
A --> 4^3 mod 17 = 13
B --> 13^5 mod 17 = 13

    Esta función genera la misma clave simétrica (número) en ambos extremos, permitiendo ya cifrar y descifrar con ella todos los mensajes de dicha comunicación.

    ¿Debilidades?¶

    • La debilidad de este algoritmo es que se pueden capturar partes de la comunicación, por ejemplo la semilla o la clave pública.

    • Es muy importante que la semilla, a partir de la cual se genera todo lo demás sea muy grande, para que aunque se capturen el resto de partes que sí viajan por Internet, no sea factible resolver la ecuación en un espacio de tiempo significativo.

    Funcionalidad

    Diffie-Hellman sólo se usa para el intercambio de claves, NO se usa para autentificar.

    Lo único que hace es generar una clave simétrica de forma segura, de tal manera que sabemos que el mensaje entre estos dos extremos es seguro.

    La autentificación se realiza a través de certificados digitales (RSA) o en su defecto usuario/contraseña.

    Integridad de los mensajes.¶

    SSH usa HMAC (Hash-based Message Authentication Codes) para comprobar la integridad de los mensajes, es decir, comprobar que nadie ha capturado el mensaje del emisor, modificado su contenido y reenviado al receptor.

    ¿Cómo funciona HMAC?
    1. Se usa en cada mensaje enviado, tanto del emisor como del receptor.
    2. El emisor junta la clave simétrica usada, el número de secuencia del paquete (TCP) y el contenido del mensaje en una misma cadena de texto.
    3. Sobre dicha cadena de texto aplica una función hash, obteniendo un código hash que se adjunta en el paquete IP del mensaje que se envía.
    4. El receptor recibe el mensaje, junta la clave simétrica (que no viaja por Internet y fue obtenida mediante DH), el número de secuencia y el contenido del mensaje.
    5. Aplica la misma función hash que usó el emisor para generar el código hash.
    6. Compara su código hash con el recibido en el paquete del emisor.
    7. Si los códigos son iguales entonces el mensaje no ha sido alterado.

    CLAVE SIMÉTRICA

    Recordad que la clave simétrica ha sido generada mediante Diffie-Hellman por lo que no puede haber sido interceptada por los atacantes.
    ¿Cómo sabe el receptor qué función hash usar?

    Cuando se establece la comunicación inicial, al igual que se informan de qué cifrado simétrico van a usar o la clave criptográfica, también se establece que tipo de función HASH van a usar.

    ¿Qué sucede si alguien intercepta el paquete del emisor antes de que llegue al receptor?

    No podrá descifrar el mensaje al no tener la clave criptográfica. No puede modificar nada del mensaje, pues a la menor modificación la clave hash generada en el receptor no casaría con la enviada por el emisor.

    ¿Y si el receptor modifica el mensaje, genera una hash con la misma función que están usando los extremos y adjunta este nuevo hash en vez del que había?

    No serviría porque para generar este código hash el receptor va a usar la clave simétrica, que no tiene la persona que ha capturado los datos, por lo que el receptor se daría cuenta de que ha habido alguna modificación.

    Funcionamiento de SSH.¶

    1. Cada extremo propone los cifrados simétricos y funciones hash que soportan y llegan a un acuerdo de qué métodos/algoritmos de cifrado y hashing usar.

    2. El servidor debe “autentificarse” (verificar que es quien dice ser), para ello se transfiere la clave pública RSA del servidor y se compara el fingerprint con el almacenado anteriormente (de otras conexiones previas).

    3. Se usa el algoritmo Diffie-Hellman para el intercambio de claves del cifrado simétrico. El objetivo de ese algoritmo es generar la clave simétrica en ambos extremos sin la necesidad de que ésta viaje por Internet.

    4. Ahora se deja el cifrado asimétrico de lado y únicamente se cifra con cifrado simétrico (normalmente AES).

    5. Finalmente queda autentificar al cliente desde el lado del servidor. Esto se puede realizar mediante el uso de certificados digitales o usuario/contraseña.

    Anexo.¶

    OpenSSH.¶

    • Conjunto de aplicaciones que permiten la administración remota de un equipo de forma segura empleando el protocolo SSH.

    • Conjunto de aplicaciones de OpenSSH:

      💥 ssh: Usado para gestión remota segura de equipos (reemplaza a telnet).

      💥 scp: Transferencia segura de ficheros.

      💥 sftp: Transferencia segura de ficheros (reemplaza a ftp).

      💥 ssh-keygen: Herramienta para inspeccionar y generar claves RSA.

      💥 ssh-keyscan: Escanea una lista de clientes y recolecta sus claves públicas.

      💥 ssh-copy-id: Facilita la copia de los certificados de acceso en los servidores.
