    PPTP (Point-To-Point Tunneling Protocolo).¶

    PPTP creado en 1999 es un protocolo VPN no seguro.

    Se intentó hacer más robusta mediante el protocolo de seguridad MS-CHAPv2, no obstante por un problema de diseño se podía conseguir reducir la seguridad de éste.

    Su uso sigue extendido en cierta forma debido a razones como:

    • Muy fácil de configurar.

    • Es compatible con la gran mayoría de dispositivos y sistemas operativos.

    • Al llevar un cifrado más ligero no realiza tanto impacto en la navegación descarga de ficheros a través de una VPN.

    • Al llevar un cifrado más ligero tampoco necesitas mucha potencia si debes mantener un servidor PPTP con muchas conexiones.

    Características:¶

    • Puerto 1723 TCP.

    • Puede hacer uso de varios protocolos de seguridad:

      💥 PAP (Texto Plano)

      💥 CHAP (Challenge-Handshake Authentication Protocol)

      💥 MS-CHAPv1 y MS-CHAPv2

    • Hace uso del protocolo GRE para establecer el túnel:

      💥 El protocolo GRE no cifra nada, lo único que hace es añadir una cabecera IP y una pequeña cabera con información del túnel GRE.

      💥 El router, por defecto (sin aplicar reglas de firewall), solo va a ver la IP origen y destino (lilas) y lo irá enrutando.

      💥 Cuando llegue al destino de la IP GRE (Lila) se desencapsula y entonces se mira la IP del túnel y se redirige a este segundo destino.

    ¿Qué pasaría si con otro protocolo pudiéramos cifrar los datos originales transportados mediante GRE, es decir, la parte amarilla?

    Podríamos transportar tráfico sin indicar quien es realmente el destinatario, sin exponer el puerto donde conecta ni los datos. Nos daría pie a poner una mini capa nueva de seguridad.

    ¿Qué puedo hacer si necesito usar un protocolo de red diferente a IP entre dos routers en redes diferentes y los routers intermedio no entienden ese protocolo y por tanto no pueden enrutar dichos paquetes?

    Una posibilidad, aunque no es la mejor solución, sería hacer un túnel GRE entre esos routers, de esta forma los routers intermedios estarían viendo el protocolo IP que sí entienden.

    PPTP con MS-CHAPv2.¶

    MS-CHAPv2 es el protocolo de autentificación, es decir los pasos que se deben seguir para asegurar la autentificación de los dos extremos, en este caso entre el cliente y el servidor PPTP.

    • Para autentificar el usuario hace uso de usuario/contraseña, pero el mensaje MS-CHAP puede comprobar esta contraseña sin necesidad de que viaje por Internet.

    • El nombre de usuario viaja en texto plano.

    • Debido a que el algoritmo de Cifrado DES es inseguro implemente una especie de 3DES, es decir, cifra el mismo mensaje de autentificación con 3 algortimos DES diferente. Cada algoritmo se encarga de cifrar una parte del mensaje, del byte 0 al 6 con un DES, del byte 7 al 13 con otro DES y del 14 al 20 con otro DES diferente.

    Pasos para establecer la conexión:¶

    1. El servidor:

      1. Manda un Challenge (SC) (16 bits aleatorios).

    2. El cliente:

      1. Genera su propio Challenge (CC) (16 bits aleatorios).

      2. Unifica en un mismo mensaje el Challenge del Servidor (SC) + Challenge Cliente (CC) + nombre usuario.

      3. Crea una cadena hash con MD5 sobre la contraseña y genera una salida de 128 bits (16 bytes).

      4. Crea una cadena hash función hash SHA1 que genera una salida de 160 bits (20 bytes).

      5. Esta cadena hash SHA1 de 20 bytes se cifra por trozos usando un total de 3 algoritmos DES.

        1. Al estar usando 3 algoritmos DES independiente, necesitamos 3 claves DES diferente, por lo que usamos el MD5 (16 bytes) para ello.

        2. Cada clave usará 7 bytes y necesitamos 3 claves, por tanto 21 bytes. Pero MD5 solo tiene 16 bytes, así que añadimos 5 bytes vacíos (ceros) al final del MD5 y así obtenemos 21 bytes. Del 1 – 7 es la clave del primer DES, del 8 al 14 del segundo DES y del 15 al 21 byte del MD5 será la tercera clave para el cifrado simétrico.

      6. Se envía al servidor tanto la salida SHA1, como la cadena SHA1 cifrada con 3DES, como el nombre de usuario en texto plano.

    3. El servidor:

      1. Conoce el SC (él lo creó y lo envió).

      2. Recibe el nombre del usuario en texto plano.

      3. Al saber el nombre del usuario también conoce su contraseña pues puede mirarlo en su base de datos. Además si aplica MD5 sobre dicha contraseña obtendrá la misma función hash que tenía el cliente.

      4. Como las claves usadas en 3DES provienen del MD5, podemos obtener las claves, descifrar el mensaje y compararlo con el SHA1 recibido para ver si es el mismo. Si es el mismo el cliente está autentificado.

    Debilidades:¶

    • El nombre de usuario viaja en texto plano.

    • La contraseña que se usa con MD5 no lleva “salt”, es decir, si la contraseña es “CiberSeguridad” esto genera siempre la misma clave hash de 128 bits. Añadir un salt quiere decir añadir un texto aleatorio a la contraseña para que el hash creado sea siempre diferente.

      💥 Esta debilidad permitiría hacer un ataque por diccionario o de fuerza bruta si se consigue la contraseña hasheada.

    • En el caso de la implementación “rara” que se hace de 3DES el tercer tramo sabemos que ha sido autentificado con una clave DES cuyos últimos 5 bytes son 0, por lo que es fácil de romper esta parte y de aquí debilita los otros dos DES restantes.

      💥 Fácil de romper.

      💥 Si obtienes el MD5 puedes romper el cifrado.

    • Haciendo un MITM podrías obtener el nombre de usuario, el SC (Server Challenge) y el Response Challenge (el hash SHA1 del SC+CC+User), por lo que probando combinaciones de 16 bits y luego creando la hash del SHA1 y comparándolo con el capturado podrías averiguar el CC.

    • MD5 tiene la debilidad que dos entradas diferentes pueden generar el mismo output, es decir, que un hacker puede crear un fichero que genere un hash idéntico al de otro fichero diferente.

      💥 Esto hace que MD5 no permita cumplir la integridad.
