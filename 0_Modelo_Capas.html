    Modelo de Capas.¶

    ¿Por qué el modelo de capas?¶

    La comunicación mediante las redes es posible gracias a los diferentes protocolos que han sido programados en los dispositivos. No obstante, si cada fabricante programara su propio protocolo (protocolo propietario), sin seguir ningún estándar universal, sería prácticamente imposible que dispositivos de diferentes fabricantes pudieran comunicarse entre si.

    Para solventar esta problemática se implementa/propone el modelo de capas, donde se especificaban diferentes niveles, cada nivel acogía múltiples protocolos, pero dichos protocolos tenían que cumplir unas normas básicas dependiendo a la capa a la que pertenecieran.

    ¿Qué es un protocolo?¶

    Un protocolo son una serie de normas/reglas que cada dispositivo debe cumplir para poder hacer cierta tarea (dependiendo del protocolo implementado).

    • Podemos aprovechar mejor el hardware de la máquina anfitriona, ya que estamos ejecutando varios "ordenadores" en uno mismo, aprovechando recursos que de otra forma es posible que no se llegaran a usar.

    • Permite ejecutar software que quizá sea incompatible con tu sistema operativo anfitrión, ya sea por un motivo de versiones o porque son sistemas operativos diferentes (Windows, Linux, MAC).

    Ejemplos de protocolos:

    💥 Protocolo ARP: Protocolo que permite a partir de una dirección IP encontrar su dirección física o MAC (capa 2~3).

    💥Protocolo IP: Sirve para llevar a cabo la comunicación a través de paquetes IP (capa 3).

    💥Protocolo TCP/UDP: Protocolos de comunicación mediante segmentos (capa 4).

    💥Protocolo HTTP: Protocolo de aplicación que permite a partir de un texto plano representar una página web a través de un navegador (capa 5)

    En el mundo de las redes cada protocolo va ligada a una capa o nivel. Es decir, que si hubiera un problema con el protocolo IP (capa 3), solo debemos de centrarnos en las tareas que se realizan en la capa 3, y la solución a dicho problema lo encontraremos con herramientas o conocimientos que afecten a ese nivel.ç

    Como hemos dicho cada capa se encarga de sus propios protocolos, aislándose de los protocolos del resto de capas. No obstante una capa siempre se comunicará con su capa inmediatamente superior o inferior, dependiendo de la dirección de la comunicación. Por ejemplo, la capa 4, únicamente podrá comunicarse con la capa 3 o con la capa 5.

    Modelo de referencia OSI.

    En un primer lugar se crea un modelo de 7 capas, totalmente teórico, que nunca llegó a ponerse en práctica, pero que sí sirvió de base a otros modelos.

    Modelo de facto TCP/IP.

    Este modelo es el que se usa actualmente en la práctica, se basa en el modelo OSI pero tiene 4 capas.

    En el modelo TCP/IP se agrupan en una misma capa el nivel físico y de enlace de datos. No obstantes nosotros sí las dividiremos y hablaremos de capa 1 (capa física) y capa 2 (enlace de datos o acceso a la red).

    NOTA

    A partir de ahora se hará referencia a este modelo (TCP/IP con 5 capas).

    Encapsulamiento.¶

    Hemos dicho que un protocolo pertenece a una capa, pero también se ha comentado que cada capa se comunica con su capa inmediatamente superior o inferior, dependiendo de si está recibiendo el mensaje o enviándolo.

    Envío de información.

    • Cuando enviamos información habitualmente se genera en una aplicación (capa 5), y dichos datos van bajando por la pila de red, es decir, pasa a la capa 4, luego la 3, luego la 2, etc…

    • En cada capa habrá un protocolo encargado de hacer algo que permita la comunicación.

    • Por ejemplo, alguna vez leeréis que HTTP (capa 5) es un protocolo que funciona mediante TCP (capa 4), haciendo uso de IP (capa 3) y a su vez de Ethernet (capa 2).

    • Cada vez que se envía información, a medida que baja de capa en capa, se va añadiendo información de cada protocolo, por tanto se van encapsulando cabeceras.

    Recepción de información.

    • Cuando recibimos información se produce el caso contrario, no lo recibe una aplicación, si no que dicha información primero llega a través de un medio (cable, fibra, aire) a nuestra tarjeta de red (capa 1), haciendo uso de ciertos protocolos como Ethernet (capa 2), IP (capa 3) y TCP (capa 4) llega finalmente al protocolo que sí puede comunicarse con la aplicación, en el caso de HTTP un navegador.

    • En este caso de recibir información, en el paso de las capas, de la 1 a la 5, se va aligerando el tamaño de los datos, es decir, se van desencapsulando las cabeceras.

    NOTA

    Obviamente dependiendo del protocolo usado, las cabeceras que se añaden a los datos son diferentes, pero siempre en cada nivel se añade o quita una cabecera.

    PDU (Protocol Data Unit).¶

    A nivel de nomenclatura, cada vez que los datos pasan por una capa con sus susodichas cabeceras van recibiendo nombres diferentes. Es decir, cuando a los datos se le añade la cabecera IP en la capa 3, se le pasa a llamar paquete, en caso de que estar en la capa 4 será un segmento (TCP) o datagrama (UDP), etc.
